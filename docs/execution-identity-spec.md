# Execution Identity Spec

Date: 2026-05-13
Status: design-only

This document defines the identity model for the future Persistent Execution Layer.

It is aligned with:

- `docs/persistent-execution-design-spec.md`
- `docs/state-boundary-contract.md`

It does **not** introduce implementation, persistence schema, or public API changes.

## Purpose and Definition

An execution is the durable, runtime-owned unit of work used to group all work that belongs to the same logical operation over time.

An execution exists to keep one stable identity across:

- initial admission of work
- queued or waiting periods
- retries
- recovery decisions
- produced artifacts
- final completion or failure

An execution is **not** the same thing as:

- a session
- a live key
- a request
- a provider attempt
- a stream
- an artifact

A session is an ownership/authentication scope. A request is a point-in-time invocation. A provider attempt is one delivery attempt. An artifact is one output/evidence object. The execution is the stable container that ties those together.

## Identity Rules

### Execution ID contract

- `execution_id` must be globally unique within the runtime.
- `execution_id` must be generated by the runtime, not trusted from a caller.
- `execution_id` must be immutable once created.
- `execution_id` must survive runtime restart through durable storage.
- `execution_id` must be opaque and carry no semantic meaning.

### Format and generation rules

- Recommended identifier class: UUID/ULID-grade opaque random identifier.
- The runtime may choose UUIDv4, UUIDv7, ULID, or an equivalent opaque identifier class during implementation.
- The implementation must not depend on human readability.
- The identifier must be safe for logs, operator tooling, and storage indexes.

### What must not be encoded into `execution_id`

`execution_id` must not encode:

- user identity
- session identity
- live key identity
- workspace identity
- artifact identity
- timestamps
- provider or model choice
- routing or fallback state
- machine identity
- hostnames
- filesystem paths
- execution state
- retry counts

The identifier is a pointer, not a transport for business meaning.

### Retry and resume rules

- Retry of the same logical execution keeps the same `execution_id`.
- Resume of an interrupted execution keeps the same `execution_id`.
- A new logical unit of work always gets a new `execution_id`.
- Multiple provider attempts or transport retries do not create new execution identities.

## Relationship Model

### Conceptual relationships

- One session may own many executions over time.
- One execution belongs to exactly one owning session.
- One live key may resolve to one current session at request time, but it does not define execution identity.
- One execution may have many request IDs over time.
- One execution may have zero or one active idempotency key contract at creation time.
- One execution may bind to zero or one workspace context.
- One execution may produce zero to many artifacts.

### Relationship summary

| Identifier | Relationship to execution |
| --- | --- |
| `session_id` | Parent ownership scope; one session may own many executions |
| `live_key_id` | Authentication linkage through session resolution only |
| `live_key` | Secret bearer credential, never execution identity |
| `request_id` | Per-request or per-attempt correlation under one execution |
| `idempotency_key` | Duplicate-prevention hint that may collapse repeated create calls into one execution |
| `workspace_id` | Optional bound workspace context for an execution |
| `artifact_id` | Output/evidence object associated back to one execution |

## Exposure Policy

- `execution_id`
  - operator-visible
  - runtime-owned
  - not part of the public API yet
- `session_id`
  - already exposed where current session flows expose it
  - remains an auth/session boundary identifier, not an execution identifier
- `live_key`
  - secret credential
  - never exposed in full in logs, docs, or operator artifacts
- `request_id`
  - operator/debug visible
  - suitable for correlation, not for durable ownership
- `idempotency_key`
  - caller/runtime coordination token
  - not a stable public identifier for execution lookup by default
- `workspace_id`
  - operator/internal until a dedicated workspace contract exists
- `artifact_id`
  - operator/internal until a dedicated artifact contract exists

## Ownership and Validation

Execution ownership is validated through the owning session and runtime authorization path.

The ownership rules are:

- each execution must have one owning `session_id`
- any request attempting to inspect, continue, cancel, or recover an execution must resolve to the same owning session unless an operator-approved recovery path applies
- live keys are only a session-resolution convenience and never bypass execution ownership rules
- resume operations must prove either:
  - same-session ownership, or
  - explicit operator recovery authority

Ownership validation must survive restart by deriving from durable execution ownership records, not only in-memory maps.

## Duplicate Prevention

Duplicate prevention is defined conceptually in this phase.

### Create-time duplicate policy

- duplicate create with the same `(session scope, idempotency key, logical operation contract)` should resolve to the same `execution_id`
- duplicate create without idempotency protection is treated as a new execution

### Important boundary

Duplicate prevention for execution creation is distinct from duplicate running prevention.

- create-time duplicate collapse is an identity concern
- duplicate running prevention is a transition and lease concern

The future implementation may use unique indexes, admission locks, or equivalent runtime controls, but that is intentionally deferred.

## Restart Durability

`execution_id` and its ownership mapping must survive runtime restart through durable storage.

Required principles:

- `execution_id` is durable truth
- in-memory execution maps are acceleration layers only
- restart must allow reconstruction of:
  - execution ownership
  - latest known state
  - related idempotency linkage
  - related workspace binding reference
  - related artifact references

If reconstruction is incomplete or ambiguous, the execution must be classified conservatively during recovery rather than silently recreated with a new identity.

| --- | --- | --- | --- | --- | --- | --- | --- |
| `execution_id` | Runtime-wide durable work identity | Runtime | Future durable execution record | Operators first; not public API yet | Immutable | Medium | Must be opaque, stable across retry/resume, and restart-safe |
| `session_id` | Auth and ownership scope | Runtime | Current session durability + future execution ownership link | Existing runtime/session surfaces and operators | Immutable for a given session | Medium | One session may own many executions |

| `request_id` | Per-request or per-attempt correlation | Runtime | Current request accounting and future transition/event correlation | Operators/debug tooling | Immutable per request | Low | Many request IDs may belong to one execution |

## Valid Identity Flows

### Flow 1: New execution from a session request

1. Caller authenticates through the current session/live-key path.
2. Runtime resolves owning `session_id`.
3. Runtime admits a new logical operation.
4. Runtime generates a new `execution_id`.
5. The execution is linked durably to the owning session.

Result:

- one new `execution_id`
- one owning `session_id`
- one or more request IDs may later attach

### Flow 2: Same request repeated with the same idempotency key

1. Caller submits the same logical create operation again.
2. Same session scope and same idempotency key are presented.
3. Runtime matches the existing execution admission contract.
4. Runtime returns or reuses the existing `execution_id`.

Result:

- no duplicate execution is created
- the execution identity remains stable

### Flow 3: Running execution interrupted and resumed

1. Execution enters active work.
2. Runtime interruption occurs before final completion.
3. Recovery logic determines the execution should continue rather than be replaced.
4. Execution resumes under the same `execution_id`.

Result:

- same `execution_id`
- new transition events
- likely new request IDs or retry attempts

### Flow 4: Separate user action under the same session

1. Caller uses the same valid session.
2. Caller starts a different logical operation.
3. Runtime admits new work.

Result:

- same `session_id`
- different `execution_id`

### Flow 5: Artifact attaches to an existing execution

1. Execution produces or claims an artifact.
2. Runtime verifies artifact evidence.
3. Runtime creates `artifact_id`.
4. Artifact record links back to the existing `execution_id`.

Result:

- same `execution_id`
- one new `artifact_id`

## Explicit Non-Goals

This spec does not define:

- the future wire format for exposing `execution_id`
- storage schema details
- exact idempotency index design
- workspace API contracts
- artifact evidence schema

Those are later-phase design or implementation tasks.
